1. PAPER POSITIONING
================================================================================

Proposed title:
Federated Double Deep Q-Network for Channel-Aware Task Offloading with
Rejection-Aware Adaptive Resource Allocation in Edge-Cloud IoT Systems

Core thesis:
Edge-cloud IoT task offloading should be trained and evaluated as a
channel-aware, QoS-aware, rejection-aware sequential decision problem under
non-IID edge zones. The proposed Fed-DDQN policy improves test latency and QoS
balance over implemented baseline policies while preserving a privacy-aligned
federated training structure. A second-stage allocator then refines resource
shares for the tasks selected for edge execution.

Core model hierarchy:
1. Stage 0: Dataset3, feature engineering, latency/rejection label generation.
2. Stage 1: Proposed Fed-DDQN for task offloading between edge and cloud.
3. Stage 2: Rejection-aware residual resource allocator for CPU, memory, and
   bandwidth after Fed-DDQN has selected edge execution.

The proposed/main model is Fed-DDQN. Other models are baselines for comparison.
The resource allocator is a secondary post-offloading module; it should not be
framed as replacing the offloading policy.

Conservative claim style:
- Say "realistic synthetic dataset" or "real-world-inspired synthetic traces".
- Do not say "real-world measured deployment data".
- Say the proposed method gives the lowest average test latency among the
  implemented trainable policies in the current v6.6 run.
- Do not claim dramatic dominance over FL-DDPG. The margin is small.
- Say Oracle is an unattainable/reference ceiling, not a trainable method.
- Say multi-seed evidence supports stability across three seeds, not exhaustive
  statistical proof across all operating regimes.
- If an ablation variant has slightly lower latency on one metric, explain that
  the proposed configuration is selected for balanced validation objective,
  rejection/SLA behavior, and research design consistency, not strict dominance
  on every scalar metric.

================================================================================
2. RESEARCH QUESTIONS
================================================================================

RQ1:
Can a zone-aware Fed-DDQN policy reduce task offloading latency under dynamic
edge, cloud, and network states compared with centralized DDQN, FL-DDPG, and
heuristic/metaheuristic baselines?

RQ2:
Does channel-aware and rejection-aware reward design produce better SLA and
rejection behavior than latency-only or less constrained baselines?

RQ3:
Does federated training across edge zones provide a practical structure for
non-IID IoT workload regions while maintaining competitive test performance?

RQ4:
How close is the learned Fed-DDQN policy to the Oracle ceiling across task
types, network regimes, and stress scenarios?

RQ5:
After Fed-DDQN selects edge execution, can a rejection-aware residual allocator
improve resource assignment quality and reduce under-allocation, waste, and
capacity violations compared with simpler allocation rules?

9. EQUATION PLAN
================================================================================

Equation group 1: Task and state definitions
- Task i: tau_i = (d_i, k_i, s_i, c_i, m_i, deadline_i, priority_i, arrival_i)
- Edge state e_j(t): CPU, memory, queue, energy, failure/degradation.
- Network state n(t): bandwidth, delay, SNR, packet loss, event flags.
- Cloud state c_l(t): queue, latency, capacity, maintenance/cold-start.

Equation group 2: Latency model
- Edge latency = transmission/access delay + queue delay + execution delay.
- Cloud latency = uplink/core network delay + cloud queue + cloud execution.
- Use notation consistent with compute_latency() in v6.6.

Equation group 3: SLA and rejection
- SLA_i = 1 if selected latency <= deadline_i else 0.
- Rejection flag = 1 for infeasible/impossible/risky cases as defined by the
  notebook labels.
- Be careful: rejection_flag is a label/risk construct, not always a literal
  runtime rejected packet.

Equation group 4: MDP
- State s_t = feature vector from task, edge, network, cloud, and QoS features.
- Action a_t in {edge, cloud}.
- Reward r_t penalizes latency, SLA miss, rejection, overload, and poor channel
  choices.

Equation group 5: DDQN target
- y_t = r_t + gamma Q_target(s_{t+1}, argmax_a Q_online(s_{t+1}, a)).

Equation group 6: Federated aggregation
- theta_global = sum_z (n_z / sum_z n_z) theta_z.
- Add FedProx regularization: mu/2 ||theta_z - theta_global||^2.

Equation group 7: Validation score
- val_score = avg_latency + 5.0 * SLA_miss_percent + 2.5 * rejection_percent.

Equation group 8: Stage-2 allocation
- base allocation = demand-proportional CPU/memory/bandwidth vector.
- residual correction = neural allocator output.
- risk projection = adjustment based on deadline, priority, SLA risk, and
  rejection-aware features.
- normalized allocation = projection onto capacity-feasible resource vector.

Equation group 9: Metrics
- average latency
- SLA compliance
- rejection rate
- edge usage
- allocator MAE
- under-allocation
- resource waste
- capacity violation
- efficiency score

10. ALGORITHM PLAN
================================================================================

Algorithm 1: Dataset and feature construction
Inputs:
- Dataset3 CSVs.
Outputs:
- features, labels, rejection flags, temporal splits.
Steps:
1. Load Dataset3 CSVs.
2. Build lookup indexes.
3. Compute edge/cloud latency safely.
4. Generate offload labels and rejection flags.
5. Engineer 30 features.
6. Apply purged temporal train/validation/test split.
7. Fit scaler on train only.

Algorithm 2: Zone-aware Fed-DDQN training
Inputs:
- train split, validation split, zone mapping, Fed-DDQN hyperparameters.
Outputs:
- best global Fed-DDQN checkpoint.
Steps:
1. Partition train tasks by edge zone.
2. Initialize global Q network and target network.
3. For each federated round, copy global weights to local clients.
4. Train local DDQN agents on zone-specific environments.
5. Apply DDQN target and FedProx regularization.
6. Aggregate local weights by weighted FedAvg.
7. Evaluate global policy on validation split.
8. Save best checkpoint and early-stop after patience.

Algorithm 3: Test-only policy evaluation
Inputs:
- best Fed-DDQN, trained baselines, Oracle labels, test split.
Outputs:
- df_compare and scenario tables.
Steps:
1. Run each policy on test split only.
2. Compute latency, SLA, rejection, edge usage, communication delay.
3. Compare trainable policies and Oracle ceiling.
4. Generate per-task-type and scenario-wise summaries.

Algorithm 4: Rejection-aware residual resource allocator
Inputs:
- Fed-DDQN edge-selected test tasks and resource targets.
Outputs:
- CPU/memory/bandwidth allocations and allocator metrics.
Steps:
1. Build demand-proportional base allocation.
2. Train residual neural allocator.
3. Apply deadline/priority/SLA-risk correction.
4. Normalize allocations to capacity constraints.
5. Compare against allocator baselines.
6. Report MAE, under-allocation, waste, capacity violation, risk, efficiency.

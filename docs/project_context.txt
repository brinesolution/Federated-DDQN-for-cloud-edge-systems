================================================================================
1. PROJECT IN ONE SENTENCE
================================================================================

This project builds a Fed-DDQN-based process scheduling and task offloading
algorithm for deciding whether IoT tasks should run on edge nodes or in cloud;
the supervised classifier, centralized DDQN, FL-DDPG-style approximation,
heuristics, Oracle, and allocator are supporting components used for comparison,
analysis, or resource estimation, not the main proposed contribution.

================================================================================
2. RESEARCH IDEA
================================================================================

IoT devices and edge nodes have limited compute, memory, bandwidth, and energy.
Tasks arrive over time and each task must be executed either:

- on an edge node:
  low communication latency, but limited resources and possible queue/failure.

- on cloud:
  stronger compute, but network delay, packet loss, cold starts, congestion,
  jitter, outages, and cloud maintenance can hurt response time.

The decision is dynamic because every timestep can change:
- edge CPU availability
- edge memory availability
- edge queue length
- edge energy level
- edge failure/degradation/isolation
- cloud CPU availability
- cloud queue/latency/current maintenance
- uplink/downlink bandwidth
- packet loss
- SNR
- network delay
- congestion/outage/jitter storm
- task type, deadline, priority, energy, security, battery, corruption

The project argues that static heuristics are not enough. A centralised RL agent
also loses the per-zone character of IoT workloads. The proposed solution uses
federated learning across location zones so each zone learns on its own local
task distribution while a global DDQN policy aggregates knowledge.

================================================================================
10. compute_latency() LOGIC
================================================================================

Implemented in modern notebooks.

Input:
One task row.

Output:
(decision, rejected, edge_latency, cloud_latency)

Decision:
- 0 = edge
- 1 = cloud

Main steps:

1. Read task's arrival_time and assigned_edge_id.

2. Look up:
   edge = edge_idx[(arrival_time, assigned_edge_id)]
   net = net_idx[arrival_time]
   c_rows = cloud_idx[arrival_time]

3. If edge/net/cloud lookup is missing:
   return cloud decision with capped edge/cloud latency.

4. If edge is failed:
   return cloud decision with capped latency.

5. If network outage:
   return cloud decision with capped latency.

6. If task is corrupt:
   return random-ish decision with capped latency and rejected = 1.

7. Filter cloud rows:
   use non-maintenance cloud rows if available, otherwise use all c_rows.

8. Compute cloud CPU and cloud base latency as averages across available cloud rows.

9. Compute effective bandwidth:
   snr_factor = clip(snr_db / 25.0, 0.1, 1.5)
   effective_bw = uplink_bandwidth * (1 - packet_loss_rate) * snr_factor

10. Compute edge latency:
    cpu_cycles / edge_cpu_available + edge_queue_length * 0.5
    Multiply by 1.35 if edge is degrading.
    Multiply by 1.20 if task is low-battery.
    Cap at EDGE_LAT_CAP.

11. Compute cloud latency:
    task_size_mb / effective_bw
    + cpu_cycles / cloud_cpu
    + network_delay_ms
    + cloud_latency_current average
    + cold start penalty if had_cold_start
    Cap at CLOUD_LAT_CAP.
    Add exponential jitter during jitter storm.

12. If impossible_deadline:
    return rejected = 1 with computed latencies.

13. Set rejected = 1 if any QoS threshold is bad:
    edge_queue_length > QUEUE_THRESHOLD
    packet_loss_rate > PACKET_LOSS_THRESHOLD
    effective_bw < BANDWIDTH_THRESHOLD
    edge_energy_level < ENERGY_FLOOR
    snr_db < SNR_THRESHOLD

14. Emergency tasks override rejected to 0.

15. If rejected:
    usually cloud is forced.
    Otherwise choose edge if edge_latency < cloud_latency, else cloud.

Current constants:
- QUEUE_THRESHOLD = 30
- PACKET_LOSS_THRESHOLD = 0.15
- BANDWIDTH_THRESHOLD = 80
- SNR_THRESHOLD = 5.0
- ENERGY_FLOOR = 50.0
- EDGE_LAT_CAP = 500.0 ms
- CLOUD_LAT_CAP = 800.0 ms

15. STAGE 2: Fed-DDQN MODEL
================================================================================

v6.3 proposed model:
QNetwork is a Dueling DDQN architecture.

Architecture idea:
- shared feature extractor
- value stream V(s)
- advantage stream A(s,a)
- Q(s,a) = V(s) + A(s,a) - mean_a(A(s,a))

This separates the value of a state from the advantage of each action.

v6.3 improvements:
- Dueling QNetwork.
- Weighted FedAvg by zone data size.
- FedProx proximal penalty to limit local zone drift.
- SmoothL1Loss / Huber loss for TD update.
- Larger batch size.
- More rounds and steps.
- Channel-aware reward.

30.2 Project Goal From Beginning To Current Version
--------------------------------------------------------------------------------

The project started as a synthetic IoT edge/cloud task offloading experiment and
has evolved into a research-paper pipeline where the proposed method is a
Federated Double Deep Q-Network for edge-cloud process scheduling and task
offloading.

The central decision is binary:
- action 0: execute task on edge
- action 1: offload task to cloud

The decision must account for:
- task type, size, CPU cycles, memory, deadline, priority, energy, security
- edge CPU/memory/bandwidth/queue/latency/energy/failure/degradation/isolation
- cloud CPU/memory/bandwidth/queue/latency/maintenance/overload/cold start
- network uplink/downlink, delay, packet loss, utilization, load, SNR, outage,
  congestion, jitter
- rejection conditions such as corrupt data, low battery, impossible deadlines,
  overloaded queues, packet loss, and insufficient bandwidth

The proposed contribution must always be framed as Fed-DDQN. Other models are
comparison/support:
- OffloadNet: supervised classifier used as a learned non-RL baseline and feature
  sanity stage.
- DDQN: centralized/single-zone RL baseline.
- FL-DDPG: literature-inspired comparison baseline implemented in this notebook
  family as a Q-style approximation, not a full actor-critic DDPG unless later
  rewritten.
- MTOSA, GTPSO, PTS-RA, JTOS: heuristic/literature-style baselines.
- Oracle: ideal ceiling, not a trainable model.
- ResourceAllocator: separate post-offloading regressor for CPU/memory/bandwidth
  fractions, not the scheduler itself.

30.12 2026-05-29 Residual Rejection-Aware Allocator Upgrade
--------------------------------------------------------------------------------

Purpose:
The stage-2 allocator was strengthened after analysis showed the old neural
allocator was too basic and lost to simple demand-proportional allocation. The
Fed-DDQN routing/offloading stage was not changed. The primary proposed model is
still Fed-DDQN for edge/cloud offloading. This update affects only the secondary
resource-allocation stage for tasks that Fed-DDQN already selected for edge.

New allocator concept:
- Fed-DDQN first decides edge vs cloud.
- For Fed-DDQN edge-selected tasks, stage 2 starts from a demand-proportional
  allocation baseline per `(arrival_time, assigned_edge_id)` contention group.
- A small neural residual allocator learns bounded corrections over that baseline.
- The residual model receives original 30 task/system features plus:
  demand baseline shares, raw demand scores, group demand pressure, group size,
  priority rank, deadline rank, deadline tightness, edge-SLA/rejection risk,
  cloud advantage, urgency, emergency/heavy/realtime/low-battery/impossible flags,
  edge queue, packet loss, congestion, outage, and jitter flags.
- The training loss combines allocation error, residual error, rejection-risk
  weighted under-allocation penalty, and light waste penalty.
